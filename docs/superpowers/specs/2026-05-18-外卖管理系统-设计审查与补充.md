## 3 需新增的设计内容

### 3.1 SSE 实时推送方案

**技术选型：** Server-Sent Events（服务端单向推送）。

**事件类型定义：**

| 事件名 | 触发时机 | 推送对象 |
|--------|---------|---------|
| `order:new` | 顾客提交新订单 | 对应商家 |
| `order:accepted` | 商家确认接单 | 对应顾客 |
| `order:ready` | 商家备餐完成 | 空闲骑手（配送池） |
| `delivery:accepted` | 骑手接配送单 | 对应顾客 |
| `delivery:arrived` | 骑手确认送达 | 对应顾客 |
| `merchant:audited` | 管理员审核商家 | 对应商家 |

**连接管理：**

- 客户端订阅 `/api/v1/sse/events?role={MERCHANT|CUSTOMER|RIDER}`，传入 JWT Token 认证
- 服务端按 user_id + role 维护连接映射
- 客户端断开后自动重连（浏览器原生 SSE 支持 `EventSource` API 自动重连）
- 心跳间隔：30 秒发送 `:ping` 注释行保活

**影响范围：**

| 文档 | 补充项 |
|------|--------|
| 需求分析 §2.2 各模块概述 | 补充 SSE 推送说明 |
| 架构文档 §6.2 API 清单 | 新增 `GET /api/v1/sse/events` 端点 |
| 架构文档 §8 核心业务流程 | 在每个状态变更节点标注 SSE 推送 |

### 3.2 通知记录表 `notification`

**用途：** 存储用户的历史通知记录，支持"未读/已读"状态管理。

**建表 SQL（新增）：**

```sql
CREATE TABLE `notification` (
    `id`          BIGINT       NOT NULL AUTO_INCREMENT COMMENT '通知ID',
    `user_id`     BIGINT       NOT NULL                COMMENT '接收用户ID',
    `event_type`  VARCHAR(32)  NOT NULL                COMMENT '事件类型: order_new/order_accepted/order_ready/delivery_accepted/delivery_arrived/merchant_audited',
    `title`       VARCHAR(100) NOT NULL                COMMENT '通知标题',
    `content`     VARCHAR(500) NOT NULL                COMMENT '通知内容',
    `ref_id`      VARCHAR(64)  DEFAULT NULL            COMMENT '关联对象ID（如订单编号）',
    `is_read`     TINYINT      NOT NULL DEFAULT 0      COMMENT '是否已读: 0未读/1已读',
    `create_time` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_user_id` (`user_id`),
    KEY `idx_user_read` (`user_id`, `is_read`),
    CONSTRAINT `fk_notification_user` FOREIGN KEY (`user_id`) REFERENCES `user`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='通知记录表';
```

**API 补充：**

```yaml
GET    /api/v1/notifications              # 获取通知列表（分页，按时间倒序）
PUT    /api/v1/notifications/{id}/read    # 标记已读
PUT    /api/v1/notifications/read-all      # 全部标记已读
GET    /api/v1/notifications/unread-count # 未读数量
```

### 3.3 操作日志表 `operation_log`

**用途：** 满足安全性需求中"记录关键操作日志，便于审计追踪"的要求，结构化存入数据库。

**建表 SQL（新增）：**

```sql
CREATE TABLE `operation_log` (
    `id`          BIGINT        NOT NULL AUTO_INCREMENT COMMENT '日志ID',
    `user_id`     BIGINT        DEFAULT NULL             COMMENT '操作用户ID（未登录操作可为NULL）',
    `user_role`   VARCHAR(8)    DEFAULT NULL             COMMENT '操作用户身份',
    `operation`   VARCHAR(32)   NOT NULL                 COMMENT '操作类型: LOGIN/LOGOUT/SUBMIT_ORDER/ACCEPT_ORDER/CANCEL_ORDER/CONFIRM_RECEIPT/ACCEPT_DELIVERY/UPDATE_DELIVERY_STATUS/MERCHANT_AUDIT/DISH_UPDATE/USER_STATUS_CHANGE',
    `target_type` VARCHAR(32)   DEFAULT NULL             COMMENT '操作对象类型: ORDER/DISH/MERCHANT/USER/DELIVERY',
    `target_id`   VARCHAR(64)   DEFAULT NULL             COMMENT '操作对象ID',
    `detail`      VARCHAR(500)  DEFAULT NULL             COMMENT '操作详情（摘要）',
    `ip`          VARCHAR(45)   DEFAULT NULL             COMMENT '操作IP',
    `user_agent`  VARCHAR(255)  DEFAULT NULL             COMMENT '用户代理',
    `create_time` DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_user_id` (`user_id`),
    KEY `idx_create_time` (`create_time`),
    KEY `idx_target` (`target_type`, `target_id`),
    CONSTRAINT `fk_log_user` FOREIGN KEY (`user_id`) REFERENCES `user`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='操作日志表';
```

**实现建议：** AOP 切面（`@OperationLog` 注解）自动记录，不侵入业务代码。架构文档 §10.1 的 AOP 日志描述基于此表落地。

### 3.5 菜品图片上传规范

**方案：** 本地文件存储。

**存储路径：**

```
/uploads/dishes/{merchant_id}/{yyyyMMddHHmmss}_{uuid}.{ext}
```

**约束：**

| 项 | 值 |
|----|-----|
| 允许格式 | JPG / PNG / WebP |
| 最大文件大小 | 5 MB |
| 访问路径 | `/static/dishes/{merchant_id}/{filename}` |
| 上传端点 | `POST /api/v1/merchant/dishes/{id}/image` |

**URL 生成规则：** 上传成功后返回相对路径，前端拼接域名展示。删除菜品时不自动删除物理文件（防止误删），由定时任务清理未关联文件。

### 3.6 购物车表建表语句补充

**问题：** 概念逻辑结构设计 E-R 图中已定义 `购物车项` 实体，架构文档前端项目结构中也有 `CartItemDTO`，但 SQL 建表章节遗漏了 `cart_item` 表。

**建表 SQL（新增）：**

```sql
CREATE TABLE `cart_item` (
    `id`          BIGINT   NOT NULL AUTO_INCREMENT COMMENT '购物车项ID',
    `customer_id` BIGINT   NOT NULL                COMMENT '顾客ID',
    `dish_id`     BIGINT   NOT NULL                COMMENT '菜品ID',
    `quantity`    INT      NOT NULL DEFAULT 1      COMMENT '数量',
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_customer_dish` (`customer_id`, `dish_id`),
    KEY `idx_customer_id` (`customer_id`),
    CONSTRAINT `fk_cart_customer` FOREIGN KEY (`customer_id`) REFERENCES `customer`(`id`),
    CONSTRAINT `fk_cart_dish` FOREIGN KEY (`dish_id`) REFERENCES `dish`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='购物车表';
```

**设计要点：**
- `(customer_id, dish_id)` 联合唯一约束：同一顾客对同一菜品只有一条购物车记录，多次加入时合并数量
- 加入购物车时校验菜品上架状态和库存充足性
- 下单成功后删除对应购物车项

## 7 其他标注（不阻塞当前迭代）

以下问题在审查中发现，但不需要在当前版本立刻解决，记录备查：

- **数据库备份策略：** 安全性需求提到定期备份，但未定义频率和保留策略
- **菜品搜索实现：** 当前仅提到"按关键词搜索"，未明确是 SQL LIKE 还是全文索引。数据量初期较小，LIKE 可接受；成熟期需考虑 MySQL FULLTEXT 或 Elasticsearch
- **配送范围：** 缺少"哪些骑手能看到哪些配送订单"的分配逻辑，当前抢单模型隐含所有空闲骑手可见所有待配送订单
- **顾客收货确认后评价引导：** 确认收货后预留跳转评价页面的埋点
