## 1 技术选型

### 1.1 后端技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 语言 | Java 17 | LTS 版本，生态成熟 |
| 框架 | Spring Boot 3.2 | 企业级微服务基础框架 |
| Web | Spring MVC + Jackson | RESTful API |
| ORM | MyBatis-Plus 3.5 | 增强型 MyBatis，减少手写 SQL |
| 数据库 | MySQL 8.0 | 关系型数据库，事务支持完善 |
| 缓存 | Redis 7.0 | 缓存热点数据、Token 黑名单、验证码 |
| 认证 | Spring Security + JWT | 无状态认证，RBAC 权限模型 |
| 密码 | BCrypt | 单向加密存储 |
| API 文档 | Knife4j (Swagger) | 自动生成 REST API 文档 |
| 参数校验 | Hibernate Validator | Bean Validation 标准实现 |
| 工具库 | Hutool 5.8 / Lombok | 常用工具，减少样板代码 |
| 构建 | Maven 3.9 | 依赖管理与构建 |

### 1.2 前端技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 语言 | TypeScript 5 | 类型安全 |
| 框架 | Vue 3 (Composition API) | 渐进式 MVVM 框架 |
| 构建 | Vite 5 | 极速 HMR 和构建 |
| UI 库 | Element Plus | 企业级 Vue 3 组件库 |
| 路由 | Vue Router 4 | SPA 路由管理 |
| 状态 | Pinia 2 | Vue 3 官方状态管理 |
| HTTP | Axios | 请求封装、拦截器 |
| CSS | UnoCSS / Scoped Less | 原子化 CSS + 组件样式隔离 |
| 工具 | @vueuse/core / dayjs | 组合式工具函数、日期处理 |

### 1.3 技术选型理由

- **Spring Boot 3 + Java 17**：国内高校和中小公司主流后端栈，参考文档丰富，Spring Security + JWT 实现 RBAC 开箱即用。
- **Vue 3 + Element Plus**：上手成本低，Element Plus 提供完整的中后台组件（表格、表单、对话框、权限指令），适合管理系统场景。
- **MyBatis-Plus**：相比 JPA 更灵活，复杂查询（统计报表多维度聚合）直接写 SQL，简单 CRUD 零代码。
- **Redis**：JWT 令牌注销黑名单、短信验证码过期、菜品/商家高频查询缓存。

### 2.2 分层架构（后端）

采用经典四层架构：

```
┌─────────────────────────────────────────────┐
│  Controller 层   — REST API 端点             │
│  ├─ 接收请求、参数校验 (@Validated)            │
│  ├─ 调用 Service 层                          │
│  └─ 统一响应封装 (Result<T>)                  │
├─────────────────────────────────────────────┤
│  Service 层      — 业务逻辑                   │
│  ├─ 业务流程编排                              │
│  ├─ 事务管理 (@Transactional)                │
│  └─ 权限校验（方法级 @PreAuthorize 注解）       │
├─────────────────────────────────────────────┤
│  Mapper 层       — 数据访问 (MyBatis-Plus)    │
│  ├─ BaseMapper 自动 CRUD                     │
│  └─ 自定义 SQL（复杂统计查询）                  │
├─────────────────────────────────────────────┤
│  Entity / DTO / VO  — 数据对象               │
│  ├─ Entity：数据库实体映射                     │
│  ├─ DTO：接口入参                             │
│  └─ VO：接口出参                              │
└─────────────────────────────────────────────┘
```

### 5.2 建表 SQL

```sql
-- ==================== 用户账号表 ====================
CREATE TABLE `user` (
    `id`          BIGINT       NOT NULL AUTO_INCREMENT  COMMENT '用户ID',
    `phone`       CHAR(11)     NOT NULL                 COMMENT '手机号（登录账号）',
    `password`    VARCHAR(128) NOT NULL                 COMMENT '密码（bcrypt加密）',
    `role`        VARCHAR(8)   NOT NULL                 COMMENT '用户身份: ADMIN/MERCHANT/CUSTOMER/RIDER',
    `status`      VARCHAR(4)   NOT NULL DEFAULT '正常'   COMMENT '账号状态: 正常/禁用',
    `create_time` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
    `update_time` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_phone` (`phone`),
    KEY `idx_role` (`role`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户账号表';

-- ==================== 顾客信息表 ====================
CREATE TABLE `customer` (
    `id`              BIGINT       NOT NULL COMMENT '顾客ID（关联 user.id）',
    `name`            VARCHAR(20)  NOT NULL COMMENT '姓名',
    `phone`           CHAR(11)     NOT NULL COMMENT '手机号',
    `default_address_id` BIGINT   DEFAULT NULL COMMENT '默认配送地址ID',
    `create_time`     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time`     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_default_address` (`default_address_id`),
    CONSTRAINT `fk_customer_user` FOREIGN KEY (`id`) REFERENCES `user`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='顾客信息表';

-- ==================== 配送地址表 ====================
CREATE TABLE `address` (
    `id`          BIGINT       NOT NULL AUTO_INCREMENT COMMENT '地址ID',
    `customer_id` BIGINT       NOT NULL                COMMENT '顾客ID',
    `province`    VARCHAR(20)  NOT NULL                COMMENT '省',
    `city`        VARCHAR(20)  NOT NULL                COMMENT '市',
    `district`    VARCHAR(20)  NOT NULL                COMMENT '区',
```

### 5.3 核心设计说明

- **ID 策略：** `user.id` 作为账号唯一标识，`customer.id`、`merchant.id`、`rider.id` 均为 `user.id` 的副本，实现 1:1 关联，同时允许各角色有独立的扩展字段。
- **价格快照：** `order_item.unit_price` 和 `order_item.dish_name` 在下单时复制当前菜品价格和名称，避免后续菜品涨价/改名影响历史订单。
- **订单编号：** 格式 `YYYYMMDD + 6位流水`（如 20260509000001），利用 Redis 每日自增计数器生成，保证全局唯一且可读。
- **参照完整性：** 所有外键均添加 `CONSTRAINT`，确保数据一致性。生产环境下若性能敏感可选择性移除外键但保留索引。
- **地址独立存储：** 配送地址从顾客表中独立为 `address` 表，支持顾客管理多个地址（家、公司等），其中一个为默认地址。下单时通过 `order_info.address_id` 快照关联。
- **通知记录持久化：** `notification` 表存储所有推送通知的历史记录，支持已读/未读状态管理。通知与 SSE 推送同步创建：推送成功则插入通知记录，用户可在通知中心回溯。
- **操作日志审计：** `operation_log` 表通过 AOP 切面自动记录关键操作（登录、下单、状态变更等），便于安全审计和问题溯源。
- **cart_item 唯一约束：** 同一顾客对同一菜品只有一条购物车记录（`uk_customer_dish`），多次加入同一菜品时合并数量而非新增记录。
- **支付预留字段：** `order_info` 新增 `payment_method`、`payment_status`、`payment_time` 三个预留字段，本期置 NULL，后续接入支付时无需改表结构。
- **商家自动接单：** `merchant.auto_accept` 标志位，为 1 时顾客提交订单后系统自动确认接单，跳过"待接单"状态直接进入"备餐中"。

### 6.3 统一响应与错误码

```java
// 统一响应体
@Data
public class Result<T> {
    private int code;
    private String message;
    private T data;

    public static <T> Result<T> ok(T data) {
        return new Result<>(200, "success", data);
    }

    public static <T> Result<T> fail(int code, String message) {
        return new Result<>(code, message, null);
    }
}

// 错误码规范
// 200     — 成功
// 40001   — 参数校验失败
// 40101   — Token 过期
// 40102   — Token 无效
// 40301   — 无操作权限
// 40401   — 资源不存在
// 41001   — 手机号已注册
// 41002   — 密码错误
// 42001   — 库存不足
// 42002   — 菜品已下架
// 42003   — 订单状态不允许此操作
// 42004   — 配送地址不完整
// 50001   — 服务器内部错误
```

## 8 核心业务流程实现

### 8.1 订单提交流程（关键事务）

```java
@Service
public class OrderServiceImpl implements OrderService {

    @Override
    @Transactional(rollbackFor = Exception.class)
    public OrderVO submitOrder(OrderSubmitDTO dto) {
        // 1. 校验配送地址
        Address address = addressMapper.selectById(dto.getAddressId());
        if (address == null || !address.getCustomerId().equals(currentUserId())) {
            throw new BusinessException(42004, "配送地址不存在");
        }

        // 2. 查询购物车菜品，校验上架状态和库存
        List<CartItemDTO> items = dto.getItems();
        for (CartItemDTO item : items) {
            Dish dish = dishMapper.selectById(item.getDishId());
            if (dish == null || !"上架".equals(dish.getStatus())) {
                throw new BusinessException(42002, "菜品【" + dish.getName() + "】已下架");
            }
            if (dish.getStock() < item.getQuantity()) {
                throw new BusinessException(42001, "菜品【" + dish.getName() + "】库存不足");
            }
        }

        // 3. 计算总金额（每条明细 = 成交单价 × 数量）
        BigDecimal totalAmount = items.stream()
            .map(item -> {
                Dish dish = dishMapper.selectById(item.getDishId());
                return dish.getPrice().multiply(BigDecimal.valueOf(item.getQuantity()));
            })
            .reduce(BigDecimal.ZERO, BigDecimal::add);

        // 4. 生成订单编号
        String orderNo = generateOrderNo(); // Redis 每日自增

        // 5. 写入订单主表
        Order order = new Order();
        order.setOrderNo(orderNo);
        order.setCustomerId(currentUserId());
        order.setMerchantId(dto.getMerchantId());
        order.setStatus("已提交");
        order.setTotalAmount(totalAmount);
        order.setRemark(dto.getRemark());
        order.setAddressId(dto.getAddressId());
        orderMapper.insert(order);

        // 6. 写入订单明细（价格快照）
        for (CartItemDTO item : items) {
            Dish dish = dishMapper.selectById(item.getDishId());
            OrderItem orderItem = new OrderItem();
            orderItem.setOrderNo(orderNo);
            orderItem.setDishId(dish.getId());
            orderItem.setDishName(dish.getName());      // 快照名称
            orderItem.setUnitPrice(dish.getPrice());      // 快照价格
            orderItem.setQuantity(item.getQuantity());
            orderItemMapper.insert(orderItem);
        }

        // 7. 扣减库存（乐观锁防止超卖）
        for (CartItemDTO item : items) {
            int affected = dishMapper.deductStock(item.getDishId(), item.getQuantity());
            if (affected == 0) {
                throw new BusinessException(42001, "库存扣减失败，请重试");
            }
        }

        return buildOrderVO(orderNo);
    }
}
```

**状态机校验实现思路：**

```java
// 定义各状态允许的下一状态
Map<String, Set<String>> ALLOWED_TRANSITIONS = Map.of(
    "已提交", Set.of("待接单", "已取消"),
    "待接单", Set.of("备餐中", "已取消"),
    "备餐中", Set.of("待配送"),
    "待配送", Set.of("配送中"),
    "配送中", Set.of("已送达"),
    "已送达", Set.of("已完成")
);

// Service 中统一校验
private void validateStatusTransition(Order order, String targetStatus) {
    Set<String> allowed = ALLOWED_TRANSITIONS.get(order.getStatus());
    if (allowed == null || !allowed.contains(targetStatus)) {
        throw new BusinessException(42003,
            String.format("订单状态【%s】不允许变更为【%s】", order.getStatus(), targetStatus));
    }
}
```

### 8.5 SSE 实时推送

采用 Spring Boot `SseEmitter` 实现服务端单向推送。

**推送事件类型：**

| 事件名 | 触发时机 | 推送对象 |
|--------|---------|---------|
| `order:new` | 顾客提交新订单 | 对应商家 |
| `order:accepted` | 商家确认接单 | 对应顾客 |
| `order:ready` | 商家备餐完成 | 所有空闲骑手 |
| `delivery:accepted` | 骑手接配送单 | 对应顾客 |
| `delivery:arrived` | 骑手确认送达 | 对应顾客 |
| `merchant:audited` | 管理员审核商家 | 对应商家 |

**连接管理器：**

```java
@Component
public class SseConnectionManager {

    private final ConcurrentHashMap<Long, SseEmitter> connections = new ConcurrentHashMap<>();

    public SseEmitter createConnection(Long userId) {
        SseEmitter emitter = new SseEmitter(0L);
        connections.put(userId, emitter);
        emitter.onCompletion(() -> connections.remove(userId));
        emitter.onTimeout(() -> connections.remove(userId));
        emitter.onError(e -> connections.remove(userId));
        return emitter;
    }

    public void sendEvent(Long userId, String eventName, Object data) {
        SseEmitter emitter = connections.get(userId);
        if (emitter != null) {
            try {
                emitter.send(SseEmitter.event()
                    .name(eventName)
                    .data(data, MediaType.APPLICATION_JSON));
            } catch (IOException e) {
                connections.remove(userId);
            }
        }
    }
}
```

**Controller：**

|------|-----|

```java
@PostMapping("/merchant/dishes/{id}/image")
@PreAuthorize("hasRole('MERCHANT')")
public Result<String> uploadImage(@PathVariable Long id,
                                   @RequestParam("file") MultipartFile file) {
    // 校验文件类型
    String contentType = file.getContentType();
    if (!List.of("image/jpeg", "image/png", "image/webp").contains(contentType)) {
        throw new BusinessException("仅支持 JPG/PNG/WebP 格式");
    }
    // 校验文件大小
    if (file.getSize() > 5 * 1024 * 1024) {
        throw new BusinessException("文件大小不能超过 5MB");
    }
    // 生成文件名和路径
    Long merchantId = getCurrentMerchantId();
    String ext = FilenameUtils.getExtension(file.getOriginalFilename());
    String filename = LocalDateTime.now().format(FORMATTER) + "_" + UUID.randomUUID() + "." + ext;
    Path dir = Path.of("uploads", "dishes", String.valueOf(merchantId));
    Files.createDirectories(dir);
    // 保存文件
    file.transferTo(dir.resolve(filename));
    // 更新菜品 image_url
    String imageUrl = "/static/dishes/" + merchantId + "/" + filename;
    dishService.updateImageUrl(id, imageUrl);
    return Result.ok(imageUrl);
}
```

## 10 开发规范

### 10.1 后端编码约定

- **分层调用：** Controller → Service → Mapper。禁止 Controller 直接调用 Mapper，禁止 Service 之间循环依赖。
- **参数校验：** 入参 DTO 使用 `@NotBlank`、`@NotNull`、`@Pattern` 等注解，不手动 if-else。
- **事务边界：** 仅在 Service 层涉及多表写入时使用 `@Transactional`，只读方法不加事务。
- **异常处理：** 业务异常统一抛 `BusinessException`（自定义 RuntimeException 子类），由 `GlobalExceptionHandler` 捕获转为统一错误响应。DA 层异常（如 SQL 错误）不应直接暴露给客户端。
- **SQL：** 简单 CRUD 用 MyBatis-Plus 内置方法，复杂查询/统计写到 XML Mapper 中，禁止字符串拼接 SQL。
- **日志：** 使用 Slf4j（Lombok `@Slf4j`），关键操作记录操作日志（AOP 自动记录方法名、参数、耗时、操作用户）。

### 10.2 前端编码约定

- **组件拆分：** 页面 = 多个业务组件 + 一个布局组件。每个组件不超过 300 行，逻辑抽到 composable。
- **API 调用：** 统一通过 `api/*.ts` 模块发起，不在组件中直接写 axios 调用。
- **路由守卫：** `router.beforeEach` 中检查 Token 有效性，无 Token 跳转登录页；根据角色过滤可访问路由。
- **权限控制：** 后端返回用户角色，前端通过 `v-permission` 指令或条件渲染控制按钮/菜单显隐。
- **错误处理：** Axios 响应拦截器统一处理 401（Token 过期跳登录）、500（服务端异常提示），业务错误码在各页面自行 toast 提示。
- **命名规范：** 组件 PascalCase，目录/文件 kebab-case，变量/函数 camelCase，类型/接口 PascalCase。

### 10.3 Git 工作流

```
main          ──●────────────────────●────────  # 生产分支（只接受 merge，不直接提交）
                 \                  /
develop          ●──●──●────●──●──●──────      # 开发主分支
                       \    \    \
feature/           ●──●  ●──●  ●──●            # 功能分支（每个功能一个分支）
  user-module
  order-module
  delivery-module
```

- **分支命名：** `feature/<模块名>`（新功能）、`fix/<问题描述>`（修复）、`hotfix/<紧急问题>`（紧急修复）。
- **Commit Message：** `type(scope): description`，如 `feat(order): 实现订单提交流程`、`fix(dish): 修复库存扣减并发问题`。
- **合并策略：** feature → develop 使用 Squash Merge 保持主干干净；develop → main 使用普通 Merge 保留完整历史。
